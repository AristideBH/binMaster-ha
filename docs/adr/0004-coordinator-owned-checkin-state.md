# Move checked_in/total_collections ownership from the sensor entity into the coordinator

Originally `checked_in`/`total_collections` lived privately on `BinMasterSensor`, restored across restarts via `RestoreEntity`. Adding a companion `switch.binmaster_*_checked_in` entity (so check-in can be a native Tile card "toggle" feature, since Tile features are domain-bound and a plain sensor doesn't get one) meant two entities needed the same mutable boolean/counter. Keeping each entity's own private copy risked them drifting out of sync (e.g. the switch shows off while the sensor's attribute still says checked_in).

Moved this state into `BinMasterCoordinator` instead — extended the cached per-bin dataclass (renamed `BinOccurrence` → `BinState`) to carry `checked_in`/`total_collections` alongside the next-occurrence fields, persisted via a dedicated `homeassistant.helpers.storage.Store` (one per config entry) rather than per-entity `RestoreEntity`. Both the sensor and the switch now just read `coordinator.data[subentry_id]` and call `coordinator.async_check_in()`/`async_set_checked_in()` to mutate it; `CoordinatorEntity`'s built-in update mechanism keeps both entities in sync for free. The midnight reset (previously a separate dispatcher signal broadcast to sensors) is now just another coordinator-internal state mutation on the same shared dict.

Trade-off: turning the switch off then on again for the same physical collection double-counts `total_collections` (turning off never decrements, to avoid punishing an accidental toggle) — accepted since that counter is a lifetime nice-to-have, not something anything else depends on for correctness.
